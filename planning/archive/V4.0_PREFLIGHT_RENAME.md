## Phase 3: Local Directory + Remote

```bash
cd ~/Developer
mv ai-editorial-assistant-v3 cardigan-editorial-assistant
cd cardigan-editorial-assistant
git remote set-url origin git@github.com:MarkOnFire/cardigan-editorial-assistant.git
git fetch origin  # verify connectivity
```

**Note on venvs:** Python venvs store absolute paths in `pyvenv.cfg` and shebang lines. If still using local venvs at this point (before Docker migration), recreate them:
```bash
python3 -m venv venv --clear && pip install -r requirements.txt
```
If Docker migration happens simultaneously, local venvs become unnecessary.

## Key Risks

| Risk | Impact | Mitigation |
|------|--------|-----------|
| OpenRouter presets out of sync | LLM calls fail | Rename both config and dashboard atomically, or defer both |
| venv breaks after directory move | Can't start services | Recreate venv (or skip if Dockerizing simultaneously) |
| Claude Desktop config not updated | MCP server unavailable | Update config + restart |
| Missed path reference | Cosmetic only | GitHub redirects cover URLs; local paths fail visibly |

---

## Coordination with Docker Migration

If executing the rename alongside `DESIGN_4.0.md` Docker work:

1. **Do Phase 1 (in-repo changes) first** — this is pure text editing, no infrastructure dependency
2. **Do Phase 2-3 (GitHub + directory rename)** — establishes the new identity
3. **Then begin Docker migration** — Dockerfiles, compose files, and CI workflows will use the new name from the start
4. **Skip venv recreation** — Docker containers handle their own Python environments

This ordering means Docker config files are never written with the old name.
