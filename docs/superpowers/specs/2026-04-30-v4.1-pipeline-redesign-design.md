# Cardigan v4.1 — Pipeline Redesign

**Date:** 2026-04-30
**Status:** Approved
**Scope:** Pipeline architecture overhaul, observability, diarization, UX polish

## Overview

Version 4.1 replaces the automatic tier escalation system with user-driven retry, swaps the manager phase for a lightweight validator, adds cost visibility, introduces a diarization service for speaker verification, removes the broken chat prototype, and closes with a comprehensive UX review.

### What Changes

| Area | v4.0 (current) | v4.1 (proposed) |
|------|----------------|-----------------|
| QA phase | Manager — prose report, forced big-brain tier | Validator — structured pass/fail per phase, auto-run, cheap model |
| Retry model | Automatic tier escalation (cheapskate → standard → premium) | User-driven: user picks model, adds notes, retries with context |
| Model routing | Preset-first with tier fallback | Phase models from Settings only, override at retry time |
| Cost visibility | None (OpenRouter dashboard only) | Per-phase actuals + pre-submission estimate |
| Duration metadata | Word-count estimate | Parsed from SRT last timestamp |
| Speaker verification | None | Optional diarization service (WhisperX container) |
| Chat agent | Broken prototype | Removed (redesign deferred to v4.2) |
| Observability | Langfuse via lazy httpx, inaccurate local reporting | Eager init, background reporting, accuracy audit |

### Out of Scope

- Chat agent redesign — deferred to v4.2 (GitHub issue to be filed)

## Section 1: Validator Phase

**Replaces:** Manager phase (prose QA report, forced big-brain tier)

**Behavior:** Runs a cheap, fast model (Haiku-class) against each phase's output with a structured checklist. Returns JSON, not prose. Auto-runs after the SEO phase completes.

**Output format:**

```json
{
  "phase_results": {
    "analyst": {"status": "pass", "flags": []},
    "formatter": {"status": "fail", "flags": ["review notes appear in transcript body"]},
    "seo": {"status": "pass", "flags": []}
  },
  "overall": "fail"
}
```

**Checklist coverage:**

- Analyst: key themes identified, segment count reasonable for duration
- Formatter: no review notes in transcript body, speaker labels consistent, no content past actual duration
- SEO: title under 60 chars, description under 160 chars, keywords present
- Overall: fail if any phase fails

**Details:**

- Uses the validator model from Settings UI (defaults to cheapest available)
- Results stored on the job record, surfaced as pass/fail indicators in the job detail view
- Failed validation does not trigger any automatic retry — surfaces the flag for the user
- No `manager_output.md` deliverable produced

**Removed:**

- Manager phase entirely (agent prompt, phase config, `FORCE_BIG_BRAIN_PHASES`)
- `manager_output.md` deliverable
- All UI references to manager phase

## Section 2: User-Driven Retry with Context

**Replaces:** Automatic tier escalation (cheapskate → standard → premium)

**Flow:**

1. Phase fails validation (or errors out)
2. Job stops. UI shows which phase failed and why (validation flags or error message)
3. User opens retry dialog:
   - Sees the previous output
   - Sees validation flags
   - Can add notes (e.g., "speaker names wrong in segment 3")
   - Can pick a different model from the dropdown (defaults to current phase model)
4. Retry runs with enriched context: previous output + flags + user notes included in the prompt
5. Validator re-runs on the retried phase

**Removed:**

- `FORCE_BIG_BRAIN_PHASES` dict
- `MINIMUM_TIER_PHASES` dict
- `max_escalation_attempts` logic in `_run_phase()`
- Tier override / forced tier logic throughout `worker.py`
- `phase_base_tiers` from config
- `duration_thresholds` from config
- `escalation` config block
- `tiers` and `tier_labels` arrays from config
- `openrouter_presets` config block
- Auto-escalation logic in retry endpoints (`jobs.py`)
- All ~45-50 stale tier/preset references across ~11 files

**Retained:**

- Per-phase model assignment from Settings UI
- Model roster (fetches available models from OpenRouter)
- `_run_phase()` method structure — simplified without the escalation loop

## Section 3: Cost Visibility

**Pre-submission estimate:**

- When a user submits a job, the UI shows an estimated cost based on:
  - Transcript token count (computed at ingest)
  - Selected model for each phase and its per-token pricing (from OpenRouter model roster)
  - Rough multiplier per phase type (formatter produces more output tokens than analyst, etc.)
- Displayed as: "Estimated cost: ~$0.14" before the user confirms submission

**Per-phase actuals:**

- OpenRouter returns `usage` (prompt/completion tokens) and `cost` in each API response
- Store per-phase: model used, input tokens, output tokens, cost
- Job detail view shows a cost breakdown table:

| Phase | Model | Tokens (in/out) | Cost |
|-------|-------|-----------------|------|
| Analyst | claude-haiku-4.5 | 12k / 3k | $0.02 |
| Formatter | claude-haiku-4.5 | 45k / 38k | $0.08 |
| SEO | gemini-flash-2.0 | 8k / 2k | $0.01 |
| Validator | claude-haiku-4.5 | 15k / 1k | $0.01 |
| **Total** | | | **$0.12** |

- Retries show as additional rows with their own costs

**Data storage:** New columns on the phase results table (or a dedicated `phase_costs` table) — `model_id`, `input_tokens`, `output_tokens`, `cost_usd`.

## Section 4: Observability Audit

**Merged with cost visibility (same sprint) — both are "know what happened" concerns.**

**Local model/phase success reporting:**

- Audit how phase success/failure is determined and stored
- Verify that the status reported in the UI matches actual phase outcomes
- Identify and fix any cases where a phase is marked "success" when it shouldn't be (or vice versa)

**Langfuse integration:**

- Rendering issues: diagnose why traces/spans aren't displaying correctly in the Langfuse dashboard. Could be malformed span data, missing parent IDs, or incorrect timing.
- Performance impact: the lazy-initialized `httpx.AsyncClient` for Langfuse caused ~60s delay when first called from a concurrent context inside `asyncio.gather`. Fix options:
  - Eagerly initialize the Langfuse client at app startup
  - Move Langfuse reporting to fire-and-forget background tasks
  - Add a timeout so a slow Langfuse call never stalls job execution
- Goal: Langfuse should be purely observational — it should never affect job performance or reliability

## Section 5: Timestamp Fix & SRT Duration

**Problem:** Two compounding issues caused the timestamp agent to produce timestamps past the actual end of the episode.

**Fix 1 — Accurate duration at ingest:**

- When a job is created and an SRT file is ingested, parse the last timestamp in the SRT to get actual duration
- Store this as `duration_minutes` on the job record
- Replaces the current word-count-based estimate that produced 33.52 minutes for an 18-minute episode

**Fix 2 — Agent reads SRT content:**

- Ensure the SRT file content is included in the timestamp agent's prompt context
- Update agent instructions: "Derive all timing from the SRT timestamps. Do not use the duration_minutes metadata field for timing decisions."
- The agent should never extrapolate timestamps beyond what the SRT contains

Both fixes are belt-and-suspenders — accurate metadata prevents the bad heuristic, and reading the SRT directly means the agent doesn't need to trust metadata at all.

## Section 6: Diarization Service

**Architecture:** Standalone Docker container running WhisperX, callable by the worker over HTTP.

**Container spec:**

- Base image with Python + PyTorch (CPU)
- WhisperX installed with a small/base model (~1.5GB)
- Simple FastAPI endpoint
- Accepts audio file (or video file to transcode)
- Returns speaker-timestamped segments with confidence scores

**API contract:**

```json
// POST /diarize
// Body: multipart file upload (audio or video)
// Response:
{
  "duration_seconds": 1088,
  "speakers": ["Speaker 1", "Speaker 2", "Speaker 3"],
  "segments": [
    {"start": 0.0, "end": 4.2, "speaker": "Speaker 1", "confidence": 0.94},
    {"start": 4.2, "end": 11.8, "speaker": "Speaker 2", "confidence": 0.87}
  ]
}
```

**Integration with pipeline:**

- Worker checks if diarization service is available before the formatter phase
- If available and the job has a video/audio source, sends it for diarization
- Diarization results included as additional context in the formatter's prompt: "Here is a speaker diarization analysis for reference. Use this to verify speaker labels in the captions."
- If the service is unavailable or no media file exists, formatter runs without it — diarization is additive, not required

**Docker orchestration:**

- New service in `docker-compose.yml` and `docker-compose.prod.yml`
- Worker connects via Docker network (e.g., `http://diarization:8000/diarize`)
- Container can be stopped when not needed — worker handles unavailability gracefully

**Scope note:** This is a test of the approach. If diarization quality is good and processing time acceptable on CPU, we can optimize later (GPU, model size, caching).
